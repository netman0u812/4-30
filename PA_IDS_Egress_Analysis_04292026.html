04 Key Findings
Critical Finding 1 — 180 Critical Signatures Not Blocking (95% Server-Side) 180 sigs · 85 code-execution · 40 brute-force ▶

180 critical severity signatures are configured as alert-only, meaning they generate telemetry but take no preventive action. Of these, 171 (95%) target server-side traffic — threats served by external internet servers back to CVS hosts. This is particularly consequential on egress traffic where CVS hosts are the recipients.

Category breakdown: code-execution (85), brute-force (40), overflow (20), sql-injection (20), info-leak (12), dos (3).

Notable critical alert-only signatures include current-year threats: CVE-2025-33070 Microsoft Windows RCE, CVE-2025-24045 Windows RDS RCE, CVE-2025-50177 MSMQ RCE, and CVE-2025-4660 Windows SecureConnector RCE — all 2025 CVEs that remain detect-only.

High Finding 2 — 244 High Severity Signatures Alert-Only 244 sigs · 82 code-execution · 49 brute-force · 44 dos ▶

244 high severity signatures are alert-only. Category breakdown: code-execution (82), brute-force (49), dos (44), overflow (36), info-leak (27), sql-injection (6).

Of the 244, 210 (86%) are server-side threats. Notable high + alert-only with 2024-2025 CVEs: CVE-2024-6387 (RegreSSHion SSH authentication brute force), CVE-2024-21307 (Microsoft RDP Client RCE), CVE-2025-27533 (Apache ActiveMQ DoS), CVE-2024-49112 (Windows LDAP RCE — critical severity, alert-only).

RegreSSHion (CVE-2024-6387) is particularly notable — it was a widely exploited SSH RCE vulnerability in mid-2024. Its associated signature being alert-only rather than blocking on an egress profile means outbound exploitation attempts to internet SSH servers would be detected but not prevented.

Critical Finding 3 — Insecure-Credentials Category: 98.9% Alert-Only (1,023 of 1,034) 1,034 total · 9 blocking · 1,023 alert-only ▶

The insecure-credentials category has effectively no blocking posture. 1,023 of 1,034 signatures (98.9%) are alert-only. Only 9 signatures block, all of which are authentication bypass vulnerabilities (GitLab default credentials, Atlassian Confluence auth bypass, Cisco Smart Licensing static credential, Apache HugeGraph auth bypass, etc.).

The 1,023 alert-only signatures in this category cover hardcoded credentials, default password usage, plaintext credential transmission, weak authentication protocols, and credential stuffing detection. On an internet egress profile this means:

→ CVS hosts transmitting credentials in cleartext outbound — detected, not blocked
→ CVS hosts using default or hardcoded credentials to access external services — detected, not blocked
→ Outbound credential theft exfiltration patterns — detected, not blocked

⚠ SOC 2 Relevance: The PBM SOC 2 audit identified credential/authentication issues in 4 separate findings (PBM-037.1, PBM-037.2, PBM-041, PBM-042). Having the IDS credential-detection layer also non-blocking compounds exposure in a credential risk landscape that is already audit-flagged.
Critical Finding 4 — Packet Capture Disabled on All 22,099 Signatures 100% of signatures · Zero forensic artifact capture ▶

Every signature in this policy — without exception — has packet capture set to disable. This means the IDS platform generates no packet-level forensic artifact when any signature fires, regardless of severity.

Operational impact: When a critical code-execution signature fires, the SOC receives an alert with metadata (signature ID, source/destination IP, timestamp, protocol) but no packet payload to reconstruct what actually occurred. Incident investigators cannot determine whether an alert represents a true positive exploit attempt, a false positive, or a probe vs. a successful exploitation.

Options PA supports:
→ single-packet: captures only the triggering packet (low storage overhead, high value for triage)
→ extended: captures additional context packets
→ disable: current state

At minimum, enabling single-packet capture for critical and high severity signatures would materially improve SOC investigation capability with minimal performance or storage impact.

⚠ SOC 2 Relevance (MWDC-013.1): EY tested that "for a sample of medium, high, or critical severity alerts, an incident ticket was created to investigate and remediate the vulnerability." Without packet capture, those investigations are conducted on alert metadata alone. This is a defensible but weaker evidentiary position — and would be visible to an auditor reviewing incident investigation quality.
Medium Finding 5 — Protocol-Anomaly and Brute-Force Categories Nearly Entirely Alert-Only protocol-anomaly: 86.1% alert-only · brute-force: 66.7% alert-only ▶

Protocol-anomaly (158 sigs, 86.1% alert-only): These signatures detect deviations from protocol standards — malformed headers, unexpected session behaviors, evasion-indicative anomalies. Having 136 of 158 as alert-only means protocol-based evasion techniques are being logged but not blocked. This category is typically kept in alert mode during initial tuning to measure false positive rates, but should not remain there indefinitely on a production internet egress policy.

Brute-force (171 sigs, 66.7% alert-only): 40 of the alert-only brute-force signatures are rated critical severity. This category includes SSH brute force (CVE-2024-6387 RegreSSHion), HTTP flooding, SCTP INIT floods, GTPv1/v2 floods, and credential stuffing patterns. Only 8 of 171 brute-force signatures actually block. The telecom protocol floods (GTP, S1AP, Diameter) in this category are likely safe to move to alert-only intentionally as they would not appear in CVS egress traffic, but the HTTP DoS and credential brute-force signatures represent a clear hardening opportunity.

Medium Finding 6 — 18 Legacy DROP Rules from 2002–2007 Era 18 sigs · All CVEs 2002–2007 · No recent coverage in drop action ▶

The 18 signatures configured as silent drop (rather than reset) all cover vulnerabilities from 2002–2007: SQL Server 2000 (CVE-2002-0729), Windows DHCP (CVE-2004-0899, CVE-2006-2372), ISC DHCP (CVE-2004-0460), KAME IKE/racoon (CVE-2004-0607, CVE-2004-0164), Check Point VPN-1 (CVE-2004-0699), OpenBSD ISAKMP, Sun Solaris in.rwhod, Citrix Program Neighborhood, Samba mailslot (CVE-2007-6015), and macOS KeyChain (no CVE).

These are almost certainly remnants of manual policy customization from the pre-Aetna merger era, possibly predating the current platform. Drop vs. reset is a deliberate choice (drop is stealthier — no TCP RST is sent to the attacker), suggesting these were purpose-set, not defaults. However their presence raises the question: if these 2004-era DHCP and VPN vulnerabilities warranted explicit drop, why do 2025 critical RCEs only get alert?

These rules are low risk to retain but should be reviewed and documented — if the affected platforms no longer exist in the environment, retirement would clean up the policy and improve auditability.

Note Finding 8 — 20.4% of Signatures Have No CVE Mapping (4,504 sigs) 4,504 sigs without CVE · behavior-based or vendor-proprietary ▶

4,504 signatures (20.4%) have no CVE associated. These represent behavior-based detections (protocol anomalies, generic shellcode patterns, traffic behaviors), vendor-proprietary signatures, or signatures covering vulnerabilities that were never formally assigned a CVE. This is expected and not inherently a gap — PA's threat intelligence includes significant non-CVE-mapped behavioral coverage.

However, for vulnerability management correlation (connecting IDS hits to the vulnerability management program), these signatures are invisible to CVE-based workflows. If your VM program uses CVE as the correlation key (Qualys, Tenable, etc.), any alert firing on a non-CVE signature cannot be automatically correlated to an asset vulnerability finding. This creates a blind spot in automated triage workflows.

05 Recommendations
Immediate
R1 — Harden Critical + High Server-Side Code-Execution to reset-server
The 99 critical server-side code-execution and overflow signatures currently on alert should be moved to reset-server action. This is the lowest-disruption blocking action for server-to-client threats — it terminates the server-side connection without affecting other sessions. Given these are PA vendor defaults, none of these signatures were specifically placed in alert mode; they simply arrived that way and have never been reviewed.
Target: 85 critical code-execution + 20 critical overflow (server-side) = 105 signatures
Action: default → reset-server override in threat prevention profile
Risk: Low false positive risk for code-execution and overflow categories — these signature types have tight pattern specificity
Prioritize 2025 CVEs first: 40172, 96029, 96517, 96514, 96579
Immediate
R2 — Enable Single-Packet Capture for Critical and High Severity Signatures
Enable single-packet capture on all critical and high severity signatures (16,334 total). Single-packet capture has negligible performance impact on modern PA hardware and provides the SOC with the triggering packet payload for every high-severity alert. This directly strengthens the investigative basis for MWDC-013.1 and closes the forensic gap for the 180 critical + 244 high alert-only signatures.
Implementation: Modify the "Alert-All" threat prevention profile — set packet-capture = single-packet for severity = critical and high
Storage estimate: Single packet typically 64–1500 bytes; at scale, minimal compared to full PCAP
No disruption to blocking posture — purely additive change
30 Days
R3 — Conduct Targeted Review of Insecure-Credentials Alert-Only Population
The insecure-credentials category's near-total alert-only posture (1,023 of 1,034) needs a structured review. Not all 1,023 need to block — some credential detection signatures are inherently high false-positive on enterprise networks (e.g., legacy protocol negotiation that looks like cleartext creds). However, given the audit-flagged credential management weaknesses across the PBM and MWDC reports, having no blocking posture in this category is difficult to justify without explicit tuning evidence.
Recommended approach:
1. Pull 90-day alert volume for this category from Splunk
2. Identify the top-20 firing signatures by event count
3. For each: determine if alert represents true positive, FP rate, and business justification for allow vs. block
4. Establish a documented exception register for any that must remain alert-only
5. Move those without a documented FP concern to reset-server
30 Days
R4 — Document and Formalize All ALLOW Action Exceptions
The 6 non-informational allow-action signatures must be formally documented as approved exceptions with named owners, business justification, and review dates. The Galil RIO PLC exception in particular needs an OT asset owner attestation. The MS IE HTTPS Proxy (CVE-2005-2830) allow is almost certainly stale given IE's retirement and should be evaluated for removal. The SMB NamePipe Write Evasion allow needs explicit justification — allowing an evasion signature is a non-trivial decision.
Output: Firewall/IDS exception register entry for each of the 6 non-informational allows
Fields: Sig ID, CVE, Business Justification, Risk Acceptance Owner, Annual Review Date
Ties to SOC 2 third-party risk and access governance control framework
90 Days
R5 — Establish a Severity-Based Blocking Policy Baseline and Tuning Process
The current profile relies entirely on PA vendor default actions. There is no documented CVS-owned policy decision about what action any severity tier should take. Establishing a formal policy baseline (e.g., "critical = block, high = block, medium = alert pending tuning, low/informational = alert") would transform this from an inherited vendor default into a managed security control with defined expectations.
Baseline proposal (starting point for discussion):
• Critical (server-side): reset-server | Critical (client-side): reset-client
• High (server-side): reset-server | High (client-side): reset-client
• Medium: alert (30-day tuning window before promotion)
• Low: alert
• Informational: alert or allow
Exceptions: documented per R4 process
Ties to semiannual firewall rule review (MWDC-013.2) — same team, same governance cadence
90 Days
R6 — Review Protocol-Anomaly and Brute-Force Alert-Only Populations
Protocol-anomaly (86.1% alert-only) and brute-force (66.7% alert-only) categories both warrant structured reviews. For brute-force, the telecom flood signatures (GTP, S1AP, Diameter, SCTP) are intentionally alert-only on a corporate egress policy and can be formally documented as such. The HTTP DoS flooding signatures (Slowloris, Torshammer), credential brute-force (MySQL, WordPress, CirCarLife SCADA), and the RegreSSHion (CVE-2024-6387) signature should be evaluated for blocking.
For protocol-anomaly: Pull 90-day firing history; categorize by FP rate
For brute-force: Segment telecom/OT signatures (legitimate alert-only) from credential/HTTP signatures (should block)
Target: Reduce brute-force alert-only rate from 66.7% to <30%
90 Days
R7 — Retire or Re-document the 18 Legacy DROP Rules (2002–2007)
The 18 silent-drop signatures are all 15–24 years old. Validate whether the affected platforms (SQL Server 2000, KAME IKE/racoon, Sun Solaris in.rwhod, Citrix Program Neighborhood, Check Point VPN-1 pre-2005) still exist anywhere in the CVS estate. If the platforms are fully retired, these rules can be removed to reduce policy complexity. If any remain, document them formally as retained exceptions. Either outcome improves auditability.
Cross-reference against ent_host_master for OS/platform inventory
Verify against CMDB (SNOW) for any active Sun Solaris, KAME IKE, or Check Point VPN-1 entries
SQL Server 2000 (CVE-2002-0729) almost certainly has no active instances — safe to retire
Process
R8 — Align IDS Policy Review to SOC 2 Semiannual Firewall Rule Review (MWDC-013.2)
The MWDC-013.2 control deviation in the current audit cycle was a missed semiannual firewall rule review. The Internet Engineering team owns that review. The IDS threat prevention profile should be incorporated into the same governance cadence — ensuring that as new signatures are added by PA content updates, their default actions are reviewed against the CVS baseline policy (R5), and any new ALLOW or anomalous alert-only assignments are flagged for review. This positions the IDS policy review as part of an existing, EY-tested control rather than a standalone process.
Integration point: Add IDS profile action distribution report to semiannual review package
Metrics to track: % blocking by severity, new sigs added since last review, ALLOW action count changes
Output: Signed-off review artifact for each of the two semiannual cycles
Owner: Network Security Engineering (already the MWDC-020.1 designated owner)
Process
R9 — Establish CVE-to-Asset Correlation Workflow for Non-CVE Signatures
The 4,504 signatures without CVE mappings create a gap in automated VM-to-IDS correlation. When these signatures fire, vulnerability management workflows (Qualys/Tenable) cannot automatically associate the alert with a known open finding on the asset. Establishing a supplemental lookup — mapping PA signature IDs to internal risk categories or platform tags — would allow the SOC to triage non-CVE alerts against the asset's known software stack rather than relying on generic categorization.
Approach: Export PA signature metadata (ID, category, platform/technology tags)
Cross-reference against ent_host_master platform inventory
Build Splunk lookup enrichment table mapping non-CVE sig IDs to technology platform categories
Enables "this host runs Apache → this sig is Apache-related → elevated triage priority"
